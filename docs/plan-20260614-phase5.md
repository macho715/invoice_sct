# SCT_ONTOLOGY — Phase 5 Reliability Baseline Plan

**Date:** 2026-06-14  
**Planning mode:** mstack-plan / Phase 1 Business Review  
**Scope:** Pre-implementation plan only. No code changes are authorized by this document.

## Phase 1: Business Review

### Current State

Phase 3 and Phase 4 have established a clean technical baseline.

- Typecheck baseline: 6 workspace projects, 0 errors.
- Test baseline: 405 tests passing.
- Security baseline: CSP, DLP output guard, SSRF allowlist, body limit, SSL verification, and timing-safe comparison are in place.
- Architecture baseline: 14 MCP tools are consolidated under `packages/tools`, and MCP server imports use the shared package source.
- Performance baseline: middleware Map eviction and rate-card batch DB query are implemented.

The remaining gap is runtime proof. The platform is tested for correctness, but it has not yet produced repeatable evidence for load behavior, DB pool pressure, or cross-service traceability.

### Target State

Phase 5 should produce a reliability baseline that can be rerun before release.

- Add load testing with k6 for the main invoice-audit path.
- Verify the DB pool limit behavior under concurrent requests, especially the 10-connection pool boundary.
- Add OpenTelemetry instrumentation across web, MCP server, and worker-relevant execution paths.
- Capture trace IDs in logs and error responses where safe.
- Publish repeatable CI or local evidence so future regressions are visible.

### Quantitative Impact

| Metric | Current | Phase 5 Target |
|--------|---------|----------------|
| Existing automated tests | 405 passing | 405 remain passing |
| Load-test coverage | 0 suites | 1 reproducible k6 suite |
| Runtime trace coverage | Ad hoc logs | Trace spans for web API, MCP tool call, DB query, and worker/export path |
| DB pool evidence | Assumed from config | Measured at 10-connection pressure point |
| Release evidence | Unit/integration tests only | Tests plus load and trace artifact |

### Options

| Option | Description | Effort | Risk | Estimated Cost |
|--------|-------------|--------|------|----------------|
| A | Load-test only. Add k6 scenarios and DB pool pressure checks. | 2-3 days | Low-medium | 0-300 AED |
| B | Observability only. Add OpenTelemetry instrumentation and trace propagation. | 3-4 days | Medium | 0-500 AED |
| C | Combined reliability baseline. Add k6 load tests, OpenTelemetry, DB pool evidence, and reporting. | 5-7 days | Medium-high | 0-900 AED |

### Recommendation

Choose **Option C: Combined reliability baseline**.

Reason: Phase 4 already closed the major correctness, security, CI, and architecture gaps. The next practical risk is production behavior under pressure. Load tests without traces can show that a slowdown exists, but they do not explain where it happens. Traces without load tests show visibility, but they do not prove capacity. Combining both creates a reusable release gate.

### Out Of Scope For Phase 5

- OAuth/JWT full auth replacement.
- `shipment` / `domestic` integration contract expansion.
- Zod to JSON Schema to Pydantic automatic generation.
- New product features.
- Database schema redesign.

These should remain candidates for Phase 6 after the reliability baseline exists.

### Rollback Plan

- Keep OpenTelemetry initialization behind environment flags.
- Keep k6 load tests non-blocking until thresholds are confirmed.
- Do not alter existing API contracts in Phase 5.
- If instrumentation causes runtime issues, disable it by environment variable and keep the existing 405-test baseline as the rollback checkpoint.

- Existing typecheck passes.

- k6 scenario runs from a documented command.
- OpenTelemetry can be enabled and disabled without code edits.
- At least one trace links API entry, tool execution, and DB call.
- The final report names measured bottlenecks or states that none were reproduced under the tested threshold.
