## 1. 시스템 개요

3-app 아키텍처로 구성된 인보이스 감사 플랫폼. 업로드된 인보이스 파일을 파싱→검증→평결→승인→익스포트하는 E2E 파이프라인.

| 계층 | 기술 | 역할 | 배포 |
|------|------|------|------|
| **Web** | Next.js 15 (App Router) | 업로드, 감사 오케스트레이션, 승인 게이트, 워크북 빌드 | Vercel |
| **Worker** | FastAPI (Python) | 파일 파싱, NotebookLM 추출, xlsx 익스포트 | Fly.io |
| **MCP Server** | Hono (TypeScript) | 14개 검증 도구 (외부 MCP 클라이언트용) | Fly.io |

**공유 패키지:** 6개 (`packages/tools`, `packages/database`, `packages/contracts`, `packages/shared`, `packages/telemetry`)

**데이터 저장소:**
- **Neon Postgres:** jobs, gate results, invoices, traces, rate cards
- **Vercel Blob:** invoice/evidence 파일, export artifacts (private access)
- **Cloudflare D1:** ontology WH status (legacy, 미사용)

**테스트 베이스라인:** 492 tests (web 157 + worker-py 149 + mcp-server 186)

## 2. 전체 프로세스 흐름 (End-to-End Pipeline)

```
[1] UPLOAD ──→ [2] PARSE ──→ [3] VALIDATE ──→ [4] GATE ──→ [5] APPROVE ──→ [6] EXPORT
     │              │              │              │              │              │
     ▼              ▼              ▼              ▼              ▼              ▼
  Vercel Blob   Python Worker   MCP Tools     Gate Bridge    Human Gate    13-Sheet .xlsx
  + Job Store   (xlsx/pdf/    (14 tools,      (PASS/AMBER/   (8 triggers,   (signed URL,
                md/txt/dsv)    7 called)       ZERO/FAILED)   6 roles)       private blob)
```

### Job 상태 머신 (12 states)

```
CREATED → UPLOADED → QUEUED → PARSING → VALIDATING → REVIEW_REQUIRED
                                                           │
                              ┌────────────────────────────┤
                              ▼                            ▼
                          APPROVED                      REJECTED
                              │
                              ▼
                          EXPORTING → COMPLETED
                              
                              (실패 시)
                          UPLOADED/PARSING/VALIDATING → FAILED
```

### 3.2 STEP 2 — 파싱 (Python Worker /v1/parse)

**진입점:** `apps/worker-py/app/routes/parse.py`

| 파일 타입 | 파서 | 출력 |
|-----------|------|------|
| **xlsx** | `openpyxl` (read-only, data-only) | `NormalizedInvoice` (header + lines) |
| **md / txt** | 텍스트 정규식 | `NormalizedInvoice` |
| **pdf** | `pdfplumber` (P3A) | `PdfParseResponse` (text_spans, tables, evidence) |
| **pdf_json** | JSON 구조 파싱 (Document AI 등) | `PdfParseResponse` |
| **dsv_waybill** | DSV 전용 정규식 파서 | `DsvWaybillResult` (waybill_no, lane, timeline) |

**XLSX 파서 상세 (`app/parsers/xlsx.py`):**
- 8개 정규 필드에 대해 필드별 5~15+ 별칭 헤더 감지 (예: description="desc","item","charge description"...)
- 3-tier fuzzy 매칭: exact → normalized(alphanum) → word-boundary substring
- currency 기본값 AED, USD/AED enum 검증
- invoice_total: 헤더 영역 실패 시 마지막 10행에서 "total"/"grand total" 검색
- 신뢰도: 0.9

**PDF 파서 상세 (`app/parsers/pdf_text.py`):**
- 10MB 초과 즉시 거부 (`PDF_TOO_LARGE`)
- pdfplumber로 페이지별 text spans + table candidates 추출
- 증거 추출: 5개 REF_PATTERNS (HVDC-xxx, BL-xxx, DO-xxx, INV-xxx, PO-xxx) → EvidenceCandidate (0.80)
- DSV waybill 자동 감지 → waybill 필드, lane, timeline events 추출
- SHPT shipment doc 매핑 (BOE/DO/DN/CarrierInvoice/DAS)

**수치 무결성 검증 (`app/validators/numeric_integrity.py`):**
- 모든 라인: `|qty × rate − amount| ≤ 0.01` → PASS, 초과 → ZERO (스키마는 AMBER이나 코드는 ZERO — 불일치)
- qty/rate 누락 라인은 스킵

**의존성:**
- `httpx` — blob URL에서 파일 다운로드
- `openpyxl`, `pdfplumber`

### 3.3 STEP 3 — 검증 (CfMcpClient + 7 MCP Tools)

**진입점:** `apps/web/src/lib/cf-mcp-client.ts`  
**호출:** `apps/web/src/app/api/invoice-audit/run/route.ts` 에서 오케스트레이션

#### 검증 파이프라인 (순차 실행)

| 순서 | 도구 | 대상 | 목적 |
|------|------|------|------|
| 1 | **FX Normalization** | Per-line | 타 통화 라인을 계약 통화로 변환 (환율 3.6725) |
| 2 | **route_question** | 1회 | 감사 질문 의미 라우팅 |
| 3 | **classify_type_b** | Per-line | 라인별 TYPE-B 분류 (INSPECTION/CUSTOMS/DO/INLAND/THC/DETENTION/STORAGE/OTHERS) |
| 4 | **check_rate_card** | Per-line | 적용 요율 vs 계약 요율 (rate_cards 테이블) |
| 5 | **check_cost_guard** | Batch (1회) | qty×rate 일치 + 표준금액 대비 변동률 ≤2% |
| 6 | **check_evidence_required** | Per-line | 필수 증빙 서류 존재 확인 |
| 7 | **check_hs_uae_compliance** | Only CUSTOMS | BOE 존재 + HS 코드 유효성 |

**오류 처리:** Per-line 도구 실패는 개별 catch (ERROR 상태로 기록, 기본값 fallback). 연쇄 실패 없음.

**신뢰도:** 고정 0.95

### 3.4 STEP 4 — 게이트 판정 (Gate Bridge)

**진입점:** `apps/web/src/lib/gate-bridge.ts`

#### 평결 체계 (PASS/AMBER/ZERO/FAILED)

| 평결 | 의미 | 익스포트 | 승인 필요 |
|------|------|----------|-----------|
| **PASS** | 모든 검증 통과 | 허용 | 불필요 |
| **AMBER** | 경미한 이슈 (분산 <500 AED) | 허용 | Cost-Control Lead |
| **AMBER** | 중대 이슈 (분산 ≥500 AED) | 허용 | Finance Manager |
| **ZERO** | 중대 위반 (중복/금지 청구) | 허용 (Rule #1) | Finance Approver |
| **FAILED** | 시스템 오류/데이터 누락 | **차단** | 재처리 필요 |

#### CostGuard Band → Verdict 매핑

```
PASS → PASS
WARN → AMBER
HIGH → ZERO
CRITICAL → ZERO
```

#### 3-way Reconciliation

```
Final Subtotal (invoice header) 
  vs Line Audit Total (Σ line.amount)
  vs TYPE-B Total (Σ type_b 분류 라인)

±0.01 허용오차

│invoice_total − line_audit_total│ > 0.01  →  AMBER (LINE_TOTAL_MISMATCH)
│line_audit_total − type_b_total│  > 0.01  →  ZERO  (TYPEB_TOTAL_MISMATCH)
모두 일치                                  →  PASS
invoice_total == null                     →  AMBER (FINAL_SUBTOTAL_MISSING)
```

**최종 평결:** 모든 라인의 최악 평결 + 증거 조사 결과 + 3-way Recon 결과 중 **최악값** 채택

|----|------|--------|-----------|

| HGT_07 | 컴플라이언스 증거 누락 | ZERO | COMPLIANCE_LEAD |
| HGT_08 | Parser confidence < 0.95 | AMBER | DOCUMENT_CONTROLLER |

#### 6개 승인 역할

```
COST_CONTROL_LEAD  — AMBER (<500 AED), rate 카드 이슈
FINANCE_APPROVER   — AMBER (≥500 AED), ZERO, FX, 대형 금액
MARINE_LEAD        — MOSB 증거
COMPLIANCE_LEAD    — 컴플라이언스 증거
WAREHOUSE_MANAGER  — 창고/핸들링 증거
DOCUMENT_CONTROLLER — 파서 신뢰도
```

**Rule #1:** ZERO 평결도 FINAL_APPROVED 익스포트 허용. 워크북 자체에 차단/미검증 항목 표시.

## 7. NotebookLM 추출 파이프라인

**진입점:** `apps/worker-py/app/notebooklm/orchestrator.py`

```
blob_url → SSRF 검증 → httpx 다운로드 → SHA-256
  → MarkItDown (MCP convert_to_markdown) → markdown → SHA-256
  → NotebookLM add_source → ask_question(EXTRACTION_PROMPT)
  → parse_extraction() (JSON 파싱, 1회 재시도)
  → HMAC-SHA256 서명 → POST callback → Vercel web app
```

**추출 필드:** doc_kind, fields {invoice_no, vendor, issue_date, currency, total}, lane, amounts, shipment_ids, document_numbers, confidence, flags, evidence

**실패 처리:**
- MarkItDown/MCP 실패 → `NOTEBOOKLM_UNAVAILABLE`
- JSON 파싱 실패 → `confidence=0.0, flags=["JSON_PARSE_FAILED"]` 스텁 반환
- Callback 거부 (4xx/5xx) → `CALLBACK_REJECTED`

**MCP Client (`app/notebooklm/mcp_client.py`):**
- `mcp` Python SDK 기반 (streamable HTTP)
- Timeout: MarkItDown 30s, NotebookLM 300s (브라우저 자동화 대응)
- SDK 미설치 시 `McpClientUnavailable` fallback

| **Worker** | MarkItDown→NotebookLM→normalize→callback | +PDF 파싱, 수치 검증, 계약 검증 | ⚠️ 범위 초과 |
| **Vercel** | callback + adapter + final audit + workbook | gate-bridge, cf-mcp-client, approval gate, workbook builder | ✅ 일치 |

**중복 로직 (평결):** `worker-py/validators/numeric_integrity.py`와 `web/src/lib/gate-bridge.ts` 양쪽에 PASS/AMBER/ZERO 의미. 전자는 라인 레벨 무결성(qty×rate), 후자는 job 레벨 3-way Reconciliation — 용도가 다르므로 실제 중복은 아님.

**Migration 미완료:** Worker의 numeric_integrity.py와 workbook_contract_validate.py는 Vercel로 마이그레이션 가능하나 아직 진행되지 않음.
