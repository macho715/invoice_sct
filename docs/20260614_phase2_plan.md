# SCT Invoice Audit Platform — Phase 2 안정화 계획

> **기준:** 2026-06-14 SWARM SCOUT 7-agent + PLAN 5-agent 진단 결과
> **전제:** Phase 1 (MVP Hardening) 완료. 388 tests PASS, 8/9 gates FULL.
> **목적:** P0 취약점 해소 + 아키텍처 정비 + DX 1.8→8.5 달성

---

## 1. Overview

SCT_ONTOLOGY Invoice Audit Platform은 3-app 아키텍처(Next.js web / FastAPI worker-py / Hono mcp-server)로 운영 중이나, SCOUT 진단 결과 보안·중복·문서·테스트 전반에 P0 갭 17건이 확인되었다. Phase 2는 이 갭을 3-wave로 해소하고 프로덕션 안정성을 확보한다.

## 3. Scope

### In Scope

- Next.js 15.1.9 → 15.3.1 업그레이드 (CVE-2025-29927)
- API 인증 미들웨어 추가 (Bearer token)
- HTTP 보안헤더 6종 (CSP, HSTS, XFO, XCTO, RP, PP)
- MCP tools 중복 해소 (web copy → shared package import)
- DB pool 단일화 (web + mcp-server → `@invoice-audit/database`)
- `pnpm-workspace.yaml` + root `tsconfig.base.json` 생성
- `workbook-builder.ts` / `job-store-pg.ts` 최소 테스트 (각 4건)
- `SYSTEM_ARCHITECTURE.md` / `LAYOUT.md` 현행화
- `CLAUDE.md` repo-level 생성
- CI workflows: release-gate 병목 해소 + 중복 제거
- CORS middleware (mcp-server)
- Rate limiting (Next.js in-memory, 60 req/min)

### Out of Scope

- shpiment/domestic 통합 계약 정의 (Phase 3)
- DB 통합 (3→1 PostgreSQL) — Phase 3
- Zod→JSON Schema→Pydantic 자동생성 — Phase 3
- OpenTelemetry / 모니터링 — Phase 3
- OAuth/JWT full auth — Phase 3
- Full E2E test suite 안정화 — Phase 3
- DLP output scanning — Phase 2.5

## 7. Risks

| # | Risk | Likelihood | Impact | Mitigation |
|---|------|-----------|--------|------------|
| R1 | Next.js 15.3.1 breaking change | LOW | HIGH | Preview deploy 먼저. rollback: git revert |
| R2 | Auth middleware blocks E2E tests | HIGH | MED | test fixture에 `Authorization` header 추가 |
| R3 | MCP tools extraction breaks runtime | MED | HIGH | 6 tools unit test를 packages/tools로 먼저 이전 |
| R4 | DB pool 단일화 시 연결 누수 | LOW | HIGH | connection count 모니터링. rollback: per-app pool 유지 |
| R5 | CI workflow 변경 시 deploy 실패 | MED | HIGH | dry-run branch에서 검증 후 merge |
| R6 | Rate limiting이 Vercel serverless에 부적합 | MED | LOW | in-memory Map은 cold start 시 리셋됨. Assumption: preview/prod에서 허용 범위 내 |
| R7 | `rejectUnauthorized: false` SSL → true 시 Neon 연결 실패 | LOW | MED | staging 확인 후 프로덕션 적용 |
